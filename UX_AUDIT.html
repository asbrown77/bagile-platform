UX Design Audit

BAgile Portal

Dashboard & Course Management Hub — Design Review
Reviewed 15 April 2026  ·  Based on portal v6.3.0 (Sprint 28)

Executive Summary

The portal has a solid foundation — clean design system, cohesive visual language, and the Calendar-as-hub approach is architecturally right. But it answers the wrong question first: the dashboard leads with five financial KPI cards before showing a single course, inverting the priority for someone whose first daily question is "what's happening with my courses today?" The Calendar page is the biggest recent improvement and delivers real workflow value, but three specific friction points hit every session — the Edit button is buried at the bottom of the side panel, the gateway "Publish →" action has the same visual weight as a plain hyperlink, and the published-gateway tick is not clickable (fixed in this sprint). Several nav inconsistencies and terminology issues have accumulated into noise. Most fixes are under an hour each.

What's Working Well

Nav

Sidebar badges are excellent

Showing pending transfer counts and at-risk course counts in the nav without being on any page is exactly the right pattern for an operator tool. At-a-glance triage before you've clicked anything.

Dashboard

Running Today & At Risk sections

Right idea, right design. Colour-coded backgrounds, pulsing status dots, direct links to course detail. The "All clear" green state is a satisfying confirmation that nothing is broken.

Calendar

Gateway checklist in side panel

Representing Planned → Partial Live → Live as a checklist with per-gateway publish buttons is immediately understandable. Publishing from a side panel without leaving the calendar is a real productivity win.

Calendar

Calendar tile design

Badge image + course code + trainer avatar + enrolment count in a tiny tile is information-dense without being cluttered. Left-border as status colour is an elegant use of a compact visual channel.

Design System

Cohesive component language

Button, Badge, Card, SlideOver, PageHeader all follow the same shape/size/colour conventions. Rounded-lg everywhere, consistent uppercase tracking for labels, brand-600 blue throughout. This feels designed, not assembled.

Modals

Add Course modal dual-purpose

Reusing AddCourseModal for both create and edit with initialValues is the right call. The Virtual/Onsite and Public/Private segmented controls are more scannable than dropdowns for binary choices.

What Isn't Working

  • 1
    Dashboard answers the wrong question first

    Five KPI cards appear before a single course. Four of the five are financial metrics. At 8am, Alex wants to know: is today's PSM running? Who's at risk? The week strip is buried below financial summaries that belong on the Revenue page. The order should be: today → week strip → at risk → KPIs. Revenue is important but it is not the reason the portal is opened each morning.

  • 2
    "Calendar" and "Courses" are two nav entries for one thing

    Sprint 28 retired /courses as a list, redirecting it to /calendar?view=list. But the sidebar still lists both as separate nav items in the same group. Worse: the at-risk badge is attached to /courses, not /calendar — so the badge lights up on the wrong nav item every single day.

    Sidebar.tsx line 49: badges["/courses"] = atRiskCourses — badge is on the wrong item
  • 3
    Calendar toolbar is overloaded and confusing

    Three distinct filter zones, a status legend that doubles as a filter, and a "Decision due" indicator that looks interactive but isn't. The Decision due red dot floats inline with the filter pills, strongly implying it can be toggled. It can't. The view toggle (Calendar/List) and status filters are the same tier of control but sit on opposite ends of the toolbar.

    calendar/page.tsx lines 1084–1150
  • 4
    Edit button is buried at the bottom of the side panel

    For a planned course, Edit is the highest-frequency action. It currently appears after scrolling past: badge + dates, status badge, decision deadline, enrolment bar, notes, gateway checklist, and external links — right next to the Cancel button. Edit and Cancel are not equally risky. Edit should be in the panel header, not at the bottom alongside a destructive action.

    calendar/page.tsx — SidePanel component, edit button rendered after all other content
  • 5
    "Publish →" has the same visual weight as "View"

    Publishing a planned course to WooCommerce is a significant, consequential action. It's rendered as text-xs text-brand-600 — identical visual weight to the "View" link on an already-published gateway. This is the most important action in the planning workflow. Publishing a course to E-commerce should be a full-width primary Button when the course is in Planned status, not a text link inside a checklist row.

    calendar/page.tsx line ~587 — "Publish →" is styled identically to "View"
  • 6
    Two status vocabularies with no explanation

    The dashboard uses: running / completed / guaranteed / monitor / at risk. The calendar uses: planned / partial_live / live / cancelled. These are different models for different things, but Alex sees both daily with no bridge between them. A course that's "at risk" on the dashboard doesn't show that signal anywhere on the calendar tile or side panel.

  • 7
    Dashboard "View all" links still point to /courses

    Post-Sprint 28, two "View all" links on the dashboard still navigate to /courses, which redirects to /calendar?view=list. The URL flickers and the behaviour is technically incorrect. One-line fix per link.

    dashboard/page.tsx lines 408 and 470
  • 8
    "draft" still exists in the Badge component

    After the team decision to use "Planned" not "draft", the statusBadge map in Badge.tsx still includes draft: { label: "Draft", variant: "neutral" }. If this is ever rendered it creates incorrect terminology. The design system should enforce the standard at the component level.

    Badge.tsx line 45 — remove or rename draft → planned

Structural Improvements (sprint-sized)

A. Dashboard: Operator-first layout

The dashboard tries to be both a daily operations console and a business overview. These have different information needs and different cadences. The KPI cards (revenue this month/YTD, students, upcoming count) belong on the Revenue page. The dashboard's sole job is operational pulse: status strip at top → week strip → alerts → at-risk cards → upcoming table. Revenue is important but it's not why the portal is opened at 8am. Moving the KPIs down (or off) changes the entire character of the dashboard from "report" to "cockpit".

B. Unified navigation: one entry for Calendar + Courses

The sidebar currently lists Calendar and Courses as separate items for what is now one destination. The correct model: single "Calendar" nav item, the at-risk badge attaches here, the Calendar/List view toggle lives inside the page (it already does). The /courses/[id] detail pages remain and are reached from within the calendar. This removes the navigation ambiguity permanently and stops the badge lighting up on the wrong item.

C. Side panel: state-driven layout

The side panel renders everything in the same neutral order regardless of course state. It should lead with "what do I do next?": for Planned courses, the primary CTA should be a prominent full-width "Publish to E-commerce" button — not a text link inside a checklist row. For Live courses, the enrolment bar should be the hero element with clear colour coding. For at-risk courses, the recommended action string (currently only shown on the dashboard) should appear here too. This is a meaningful UX rewrite of the panel, probably half a sprint, but the daily-use impact would be significant.

D. Calendar: "Next up" context strip

When navigating between months, there's no persistent anchor to where you are in the actual schedule. A one-liner above the calendar — "Next course: PSM in 3 days · Mon 20 Apr" — gives instant orientation without any clicks. This is especially useful after scrolling forward to look at future months and needing to re-orient.

E. Payment visibility (when Xero re-integration lands)

There is currently no way from the portal to see whether a booking has been paid. When the Xero integration is re-enabled, payment status should be a column in the course attendee table and a signal in the side panel — not buried on a separate finance page. An unpaid booking from three weeks ago is an operational issue, not just a financial one.
